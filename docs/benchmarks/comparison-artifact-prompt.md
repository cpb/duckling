# Regenerating the benchmark comparison artifact

This is a saved **prompt**, not a description of a past run. Paste it into a
Claude Code session in this repo whenever:

- a new version's `docs/benchmarks/<environment>/<version>.json` lands (e.g.
  `0.2.1-rc1` graduates to `0.2.1`, or `0.2.2` gets its first data point), or
- `0.2.1-rc1` itself gets re-benchmarked (numbers change) before it ships.

It does **not** replace `README.md` in this directory — that file is
auto-generated by `bundle exec rake benchmark:record` from the raw JSON and
must never be hand-edited. This prompt produces a separate, richer,
one-off interactive HTML artifact for reviewing *what changed* between
versions, which the auto-generated README (latest-per-environment only,
no cross-version diff) doesn't show.

## The prompt

> Build an interactive HTML artifact comparing duckling's benchmark history
> across environments, sourced from `docs/benchmarks/<environment>/<version>.json`.
> Load the **dataviz** and **artifact-design** skills before writing any chart
> code or HTML.
>
> **Data**: read every `docs/benchmarks/*/*.json` file present — don't
> hardcode a version list or a version *count*. Group by environment, order
> versions per environment with `Gem::Version` (handles `-rc1`/`.pre.rc1`
> prerelease ordering correctly — see `benchmark/report.rb`'s
> `latest_per_environment`). Build every chart/table as a version-over-time
> trend that scales to however many versions a given environment actually
> has — 1, 2, 5, whatever `docs/benchmarks/<environment>/*.json` glob
> returns right now. Don't gate structure on a fixed count (no "if 3+
> versions, ask the user" branching): an environment with only 1 recorded
> version just renders as a single point/row with no delta, one with 6
> renders a 6-point trend line, and none of this should require touching
> the artifact's code when a 4th, 5th, etc. version is recorded later.
> Where a chart type is inherently a two-state comparison (e.g. the
> diverging %-change-by-scenario bar chart), anchor it to the latest vs.
> the immediately preceding version for each environment — not a
> user-picked pair — since that's the transition someone re-running this
> prompt after a new version lands actually wants to see.
>
> Embed the raw JSON verbatim in a `<script type="application/json">` block
> and compute every derived number (percent change, log-scale positions, GC
> deltas, etc.) in JS from that single source of truth — never hand-transcribe
> numbers into the page.
>
> **Structure** (adapt — don't force a section that has nothing to say):
> - Concurrency stat tiles: 10-thread scaling efficiency as a per-environment
>   trend line/sparkline across every recorded version, not just a single
>   before → after pair — the #64 dispatch change is one point on that line,
>   not the whole story once more versions accumulate.
> - A diverging bar chart of % change in ips by scenario, grouped by
>   environment, comparing each environment's latest version against its
>   immediately preceding one (see "Data" above). Exclude `camping_trip_email`
>   from this chart (it's ~5 orders of magnitude slower than the rest and
>   only has 3-5 iterations — high variance, wrong scale for a shared axis)
>   but keep it in the full table with a footnote.
> - A full comparison table: environment × scenario × (ips, µs/call,
>   objects/call, minor GC), with one column-group per recorded version (not
>   hardcoded to 2) so a version added later just adds a column-group. This
>   also serves as the accessibility "relief" for any categorical color that
>   falls under the 3:1 contrast floor.
> - A dispatch-overhead chart (native vs. thread-per-call, log scale) —
>   **only** using scenarios/versions that actually have `native_ips` /
>   `thread_overhead_pct` fields. That schema only exists from the #64
>   dispatch-schema version onward; guard for its absence, don't assume every
>   version has it.
> - A GC/allocation-anomaly callout — **only if the data actually shows one**.
>   Compute the callout's numbers from the data at render time (min/max
>   across scenarios, across every recorded version, not just the latest
>   pair); don't hardcode a sentence carried over from a prior run.
>
> **Environment colors**: keep `github-actions` / `claude-code-web` /
> `local-3.3` / `local-3.4` / `local-4.0` in that fixed order (issue #76
> split `local` into per-Ruby-minor buckets, since a dev machine's Ruby
> version drifts over time in a way CI runners' doesn't), using validated
> categorical hues from the dataviz skill's default palette — 5 environments
> now needs a 5-hue set, not the earlier blue/aqua/violet 3-hue set. Copy hex
> values exactly from `references/palette.md` or the validator's own output —
> never hand-type a "close enough" hex. Run
> `node scripts/validate_palette.js "<hex,hex,hex,hex,hex>" --mode light` and
> again `--mode dark` before shipping; fix anything that FAILs (a WARN on
> contrast is fine only if the table/legend "relief" is present). If another
> environment slot ever appears, extend with the next fixed-order slot and
> re-validate the new set — don't eyeball it.
>
> **Verify before calling it done**: if a connected browser tool is
> available, load the artifact and actually look at it — check tooltips,
> both themes, and layout at a narrow width. If no browser is connected, at
> minimum run the page's script in a `jsdom` sandbox (`npm install jsdom` in
> the scratchpad dir) and assert the expected element/bar/row counts, so a
> DOM-target bug (e.g. axis labels appended to the wrong container) doesn't
> ship silently. Say explicitly that jsdom only catches runtime errors and
> structure, not layout, if that's all you were able to check.

## Known gotchas (from the first run, 2026-07-04)

- `"0.2.1-rc1"` is a valid `Gem::Version` (normalizes to `0.2.1.pre.rc1`) and
  sorts correctly above `"0.2.0"` — no special-casing needed.
- The `native_ips` / `native_microseconds_per_call` / `thread_overhead_pct`
  scenario fields were added by the #64 dispatch schema; any version
  recorded before that lacks them. `benchmark/report.rb`'s own
  `build_dispatch_section` already guards this the same way
  (`scenarios.select { |s| s[:native_ips] }`) — mirror that, don't assume.
- Hand-typing "close enough" hex values for the categorical palette is an
  easy way to silently drift off the validated set — copy exact hex, then
  re-run the validator on the exact final trio/set used.
- When positioning axis-tick labels absolutely, append them to the dedicated
  axis-column element (`position: relative`, height inherited via
  flex-stretch from the sibling plot element), not to the plot's parent — an
  easy bug that puts labels in the wrong spot with no error thrown.
